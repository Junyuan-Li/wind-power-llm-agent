本文档旨在对项目背后涉及的具体算法原理、平台架构选择与微调机制进行深度技术说明，为后续二次开发或面试问答提供背书。

## 1. 核心算法设计与特征工程

### 1.1 深层时序预测网络 (LSTM Executor)
传统的纯统计学预测在遇到“风速极限突降（大风切出）”或“风向瞬间偏转”等非线性事件时，容易产生巨大偏差甚至输出异常 `0.0 kW` 或负值。本项目的预测引擎：
*   **网络架构**：采用 `Input(86) -> LSTM(64_H, 2_L) -> Dropout(0.2) -> Linear(1)`。
*   **双尺度数据标准化**：引入了严格的 `StandardScaler` 闭环。在模型反序列化时，除了读取 `state_dict`，还严格恢复训练集的方差/均值分布，保障了运行时（Inference）与训练环境同位。

### 1.2 86 维衍生特征工程矩阵
不再只依赖单纯的风速/温度/气压，而是主动构造“具备物理先验”的特征向量，增强模型的非线性捕获能力：
*   **高阶风速项**：基于风能公式 $P = 1/2 \cdot \rho \cdot A \cdot v^3$，我们在特征层构造了 $v^2, v^3$ 维度。
*   **交互项**：风速与高空空气密度的相乘交互。
*   **滚窗统计 (Rolling Window)**：近 3 小时与 6 小时的风速方差，提供“风力不稳定性/湍流指数”先验。
*   **差分项 (Diff)**：一阶与二阶差分表征天气剧变情况。

## 2. 三层 RAG（检索增强生成）引擎

本项目摒弃了“把文档直接扔给大模型”的粗暴做法，构建了结构化的向量溯源图谱：
*   **特征编码（Embedding）**：采用 `bge-small-zh-v1.5`，该模型在中文专业文献向量表征有着优秀的指标。
*   **向量库技术引擎（Vector DB）**：离线采用 Facebook 的 `FAISS`，保证了极低的内存开销和本地毫秒级的欧氏距离/内积召回。
*   **Layered Retrieval 设计**：
    1.  *理论层*：匹配当前风速所在的气动理论（如切入区、额定区、切出区）。
    2.  *历史对比层*：基于相似气象数据，查找数据库中过去的有效功率样本作为“基座锚点”。
    3.  *逻辑模板层*：拉取“排障标准作业流程 (SOP)”。

## 3. 面向本地部署的微调与推理 (LoRA & Qwen)

### 3.1 QLoRA 高效参数微调
由于本地算力限制（如只有不到 16GB 消费级显存），全量微调并不可行：
*   **技术栈选择**：采用 HuggingFace 的 `PEFT` 模块对 `Qwen/Qwen2.5-7B` 进行 8-bit 或 4-bit 注入（LoRA）。
*   **超参数组**：Rank ($r = 8$ / $16$), $\alpha = 16$ / $32$。这保证了在不破坏大规模通用常识的前提下，大模型对风电“异常诊断、功率波动的专业词汇”呈现高度敏感。

### 3.2 Ollama 本地架构
为了脱离 Python 中笨重的 `transformers.pipeline` 并提升推理流（Streaming）速度：
*   我们选用了 Ollama 框架：其底层基于 llama.cpp 对模型权重做量化执行，降低推理显存门槛，并实现与前端 Web 丝滑的逐字打印效果。

## 4. Agent 范式体系

本系统并非一个“带前端壳子的脚本”，而是一个完整的 **AI Agent (智能体)**闭环。
*   **Planner**：解析用户的中英文混合 Query，决定执行链条（例如用户只问问题，就不必去算 LSTM）。
*   **Executor**：无缝链接本地计算组件（LSTM 预判或 FAISS RAG）。
*   **Writer & CoT**：利用 **Chain-of-Thought（思维链）** 整理结果，先阐述现象，再给出物理公式原理判定，最终得出诊断。确保系统的极度严谨。
